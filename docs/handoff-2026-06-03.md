# ⍺lphaβeta — Decisions & Context Handoff for Claude Code

> **Purpose:** This document captures the design decisions, architectural commitments, and open questions from recent planning sessions (May–June 2026) for the ⍺lphaβeta project. It is intended to be read by Claude Code alongside the codebase, `CLAUDE.md`, and `TODO.md` as context for the phase-three build. It is not a task prompt — it is a reference document.
>
> **What this project is:** A serverless, browser-based discipline layer for empirical work — a decision-logging and process-management tool with experimentation as the first methodology module. Not a stats tool. It surfaces the process tradeoffs existing experimentation platforms leave implicit.
>
> **Where it lives:** `~/Projects/alphabeta/` (single-app repo, [`mattpolicastro/alphabeta`](https://github.com/mattpolicastro/alphabeta), public). The handoff's original `experiment-tools` monorepo proposal is deferred — single-app for now; revisit when the Chrome extension is ready to land alongside the app.

## 1. What's being built (the reframe)

The project began as a browser-based A/B test analysis tool (stable at v0.2.5, March 2026, now archived as [`mattpolicastro/alphabeta-legacy`](https://github.com/mattpolicastro/alphabeta-legacy)). That framing has been superseded. Most of the v0.2.5 build is being set aside. Two things carry forward into phase three:

- **The experiment schema** (data model for experiments, variations, metrics, results)
- **Reusable analysis utilities** (stats transforms, gbstats integration, power calculator)

Everything else is a ground-up rewrite oriented around a **single-bet lifecycle** — the discipline layer's core loop. See [`/reuse/README.md`](../reuse/README.md) for the full manifest.

### The positioning

- **"Can" vs. "should."** Most tools answer whether you *can* run a test. ⍺lphaβeta owns whether you *should*.
- **Not a stats authority.** It's a process-and-application authority. The stats engine is infrastructure, not product surface.
- **Serverless by design** — no accounts, no auth, no data leaving the browser in the default tier. All compute in Pyodide/WASM, all storage in IndexedDB. Privacy as a feature, not a limitation.
- **The "bets" vocabulary is deliberate.** Borrowed from John Cutler's North Star Framework (Amplitude), where strategic decisions are framed as leveled bets (Level 0 = North Star, Level 1 = Inputs, Level 2 = Initiatives). The term centers commitment under uncertainty rather than scientific testability. The wager, the fold-if, the lock — these are commitment mechanisms, not hypothesis-testing formalism. This vocabulary is load-bearing across the three-layer toolchain (Plinth Board uses it at the strategy level; ⍺lphaβeta inherits the unit).

## 2. The single-bet lifecycle (five screens)

The converged product design, arrived at through 10 wireframe iterations. The canonical wireframes live in [`/design/`](../design). The lifecycle is one carried thread, end to end.

### Screen 1: Bet Front Door
Articulate a loose idea as a falsifiable bet.
- **Flow:** Dump (natural language, say what you're thinking) → Reflect (the tool pushes back, scans for a falsifier, flags gaps) → Sharpen into the wager sentence.
- **Natural-language first.** The wager sentence is the *destination*, not the entry point. Demanding structure upfront is too heavy for first contact.
- **Theory/mechanism field:** Separated from confidence — "if / then / because." The "because" is the mechanism; it's what makes the bet falsifiable rather than just directional.
- **Express lane:** For experts who can write the wager sentence directly. Visible but not the default path.
- **Tweaks panel:** Pushback tone, falsifier insistence, express-lane visibility, accent color. **Critical constraint: the Tweaks dial settings must themselves be committed, visible, locked fields** — they are part of the lock, not exempt from it. The diagonal user (deep domain expertise, stats-shaped gap) is both the one most tempted to dial down falsifier insistence and the one most likely to be pushed around by a PM, so the knob can self-defeat the mission for exactly the population it protects.

### Screen 2: Feasibility & Instrument
The bet's fold-if becomes the detection spec; pick the instrument.
- **Carries the committed bet** from Screen 1 (via storage). The fold-if is displayed read-only (step back to Screen 1 to edit).
- **Key bridge:** If a test can't *see* the fold-if, it can't resolve the bet. The fold-if stated at the front door presupposes measurement capability — there's a latent loop here.
- **Method settles at feasibility, not before.** The user may arrive planning an A/B test; constraints (can you randomize? traffic, urgency, claim strength) can argue them off it. Methods: A/B, quasi-experiment, observational, holdback.
- **Nudge, not gate.** Overriding the recommendation is allowed but logged with a reason.

### Screen 3: Decision Criteria
Pre-register win / inconclusive / loss actions.
- **Fold-if is the loss line AND the minimum mind-changer** — one number, declared once, carried from Screen 1.
- **Evidence bar** is supplied by the chosen instrument (from Screen 2).
- Pre-register what you will *do* in each bucket, not just what the bucket thresholds are.

### Screen 4: Commit & Lock
Two-step draft / lock.
- **Draft:** editable, revisable, no timestamp.
- **Lock:** timestamps and freezes the whole pre-registration. Content fingerprint (hash) proves what was committed before any result existed.
- **Immutable.** Edits after lock create a new version, never overwrite. The fingerprint chain is the tamper-evidence mechanism — defensible with no server.

### Screen 5: Revisit
Commitment vs. outcome — the loop closes.
- **Bucket is computed from the fold-if** (not post-hoc mood). The pre-registered action is recalled.
- **Deviations are logged** in the user's words beside the timestamp. Did you follow through on what you committed to? If not, why?
- **Method-agnostic.** It only asks whether you honored the lock.
- **Calibration feeds the next bet** — expected vs actual outcomes. (Note: this is promised value, not yet live. Cross-bet memory / journal index is an open thread. Don't oversell this in UI copy.)

## 3. Core architecture principles

These emerged from the wireframe arc and are settled. Do not relitigate.

- **Fold-if as single thread.** One declared number is the detection spec, the loss line, and the bucket computation. This is the load-bearing mechanism that defeats goalpost-moving. Everything else (lock, fingerprint, deviation log) is enforcement scaffolding around it.
- **Fixed spine, variable narration.** One skeleton; the *register* (Scaffolded / Standard / Ambient) only changes how much the discipline layer talks. The spine never moves. This maps to expertise level, not workflow complexity.
- **Method as pluggable module.** The decision journal is the spine; the method (A/B, quasi-experiment, observational, holdback) is swappable. The decision is decoupled from the method. Only the feasibility surface and running discipline change when the instrument changes.
- **Accountability through visibility, not compliance gates.** Gates get routed around. Rigor comes from holding users to prior commitments and making implicit tradeoffs visible. Low-friction intent capture upfront; enforcement through what was committed, not blocking workflows.

## 4. Known design issues & open questions

From arc-level review (documented in vault at `Projects/AlphaBeta/Iteration Log — Review Notes.md`):

### `[!]` Fold-if backward-edit loop (CONTESTED — reopens a converged decision)
The fold-if flows forward from Bet Front Door → Feasibility → Criteria → Revisit. But a fold-if named at the front door already presupposes an instrument (you can't state a mind-changer in units you can't measure). When feasibility reveals the instrument can't see the fold-if, the user either revises the fold-if (good) or fudges it to fit (defeats the integrity goal). **The backward edit from Feasibility to Front Door needs to be a first-class, low-friction loop with the revision itself logged** — not a friction-heavy exception. The current "read-only fold-if, step back to edit" is under-specified.

### `[?]` Journal-index scope (OPEN — a fork not yet taken)
The journal index (decision journal, pluralized — a list of all bets and their states) is proposed as "the hub" for three jobs: input from strategy (Plinth Board seam), planning/sequencing, and KM/cross-bet learning. These have different data shapes. Current lean: index owns memory/search only; sequencing stays with Plinth Board; intake belongs to the unbuilt decision substrate. Don't over-centralize.

### `[ ]` Running-phase integrity boundary (POSITIONING — change the claim, not the build)
The current lifecycle delivers **per-bet integrity**: intent locked, outcome checked against it. It does NOT guard how the outcome number was generated (peeking, early stopping, p-hacking) — that happens between lock and revisit, and revisit is only as honest as the figure typed in. The **Chrome extension** (XHR interception of the platform's actual results stream) is the piece that closes this gap by turning the outcome from "what the PM says" into "what the platform returned." State this boundary explicitly in UI and docs: current build = per-bet integrity; extension = process-of-measurement integrity.

## 5. Hosting architecture & deployment tiers

Settled in the June 3 architecture discussion. Not yet in the vault (MCP failure during write).

### The canonical store model
- **App origin is the single canonical store.** All decision-journal data lives in the app's IndexedDB.
- **Extension is a satellite.** It keeps a small outbound queue in its own storage and reconciles into the app's IndexedDB via `externally_connectable` whenever the app context is reachable (open tab, or offscreen document).
- **Capture never blocks on the app being open.** Analysis and revisit always read one store. Core loop needs zero server.

### Three deployment tiers (same app, different backend config)

| | Static (no backend) | Self-hosted container | Hosted freemium |
| --- | --- | --- | --- |
| Operator | nobody | the user / team | you (Matt) |
| LLM key | in browser (BYO) | in container | your infra (metered) |
| Rubric | exposed (client JS) | hidden (server-side) | hidden (server-side) |
| Parse | local model | container proxy | hosted proxy |
| Feedback | BYO-key direct | container proxy | hosted proxy |
| Accounts | none | none | yes (quota) |
| Backup | file export/import | container storage | hosted sync |

- Left = more private / user owns it. Right = more convenient / rubric hidden.
- **Self-hosted and hosted are the same artifact** — one self-hosted, one run by you with auth + quota in front. Write one backend.
- **Static is fully serverless and privacy-maximal** but cannot keep the rubric secret. That's its one real limitation, specifically a *feedback* limitation.
- **Migration story:** Build static-first (complete, serverless: parse local + feedback BYO-key). Ship the container as pure upgrade. No tier is a rewrite of another.

### The provider-adapter seam
- App talks to a `LLMProvider` adapter interface and does **capability negotiation at boot** ("what am I wired to, what can I light up?").
- One interface, multiple implementations: local model / BYO-key direct / backend proxy.
- Accounts are a property of one backend, never something the app requires.
- **`LLMProvider` interface surface still to be sketched** — this is an open design task.

### Two LLM engagement levels
- **Parse (light):** Mechanical extraction of free-text intent into schema fields. Local-model viable. No rubric to protect — nothing to game. Portable across all tiers.
- **Feedback (judgment):** The "can vs should" coaching — the discipline, operationalized. Needs a strong model (local too weak). The *only* consumer of the rubric.

### The rubric (the sensitive IP asset)
The rubric = the system prompt + templates encoding what counts as rigor. Examples: does the metric actually move if the hypothesis is true; is the MDE realistic for their traffic; are they pre-registering a decision rule or leaving room to pick the favorable cut; does the success criterion match the upstream goal; thresholds for warn vs hard-flag.
- It's the genuinely hard-to-copy IP.
- **Adversarial bite:** A PM who can read the rubric can game it (sees "flag if MDE > 20%" → writes 19%). The rubric only works if the people it evaluates can't see the answer key.
- The concern is narrow: **feedback only.** Parse has no rubric to protect.
- **Self-hosted is the threat-model sweet spot.** The disciplined core team operates the container (rubric server-side); the PMs being policed are users who can't inspect it.

### Lambda fallback
The existing Lambda fallback (silent server-side stats compute when Pyodide fails) is **in tension with the privacy value proposition.** "All compute on-device" and "silently ships experiment data to a server" don't coexist. Options: consent gate before server compute, or "download a script to run locally" degrade path. Do not leave this as invisible server compute.

### Backup tiers (along the same axis)
- File export/import (floor, always serverless)
- BYO remote target (R2/S3/WebDAV/gist — user owns durability)
- Hosted sync (freemium hook)

### Tamper evidence without a server
Hash-chaining the committed fields gives tamper-evidence locally. The fingerprint chain is defensible without a server — immutable pre-registration, timestamped, verifiable.

## 6. Chrome extension architecture

Exists as a concept doc in the vault (`Projects/AlphaBeta/AlphaBeta Chrome Extension Concept.md`). Key decisions:

- **UI surface:** `chrome.sidePanel` API (Chrome 114+), persistent sidebar
- **Three roles:** Passive extraction monitor, augmentation/correction layer, quick diagnostics (SRM check)
- **What it is NOT:** A mini version of ⍺lphaβeta. No analysis in the sidebar. Its job ends at "here's what I captured, ready to send?"
- **Handoff:** `externally_connectable` to the app's custom domain. Extension calls `chrome.runtime.sendMessage(EXTENSION_ID)`.
- **Scraping strategy:** Per-platform scrapers exporting a common `ScrapedExperiment` interface.
  - GrowthBook: XHR interception of `/api/v1/experiments/{id}/results` (not DOM scraping). GrowthBook's monorepo (`packages/shared`) provides typed API contracts.
  - Adobe Target: Network interception of `performance.at.json`.
  - A4T: Deferred to v2.
- **Manifest V3.** Permissions: `activeTab`, `storage`, `sidePanel`. Host permissions for GrowthBook + Adobe Target domains. Self-hosted GrowthBook domains configurable via options page + `chrome.permissions.request()`.

**Note on prior art:** The archived legacy repo carries a `feat/extension-scaffold` branch with 8 commits of GrowthBook XHR interception, ScrapedExperiment type, message protocol, content scripts (MAIN + ISOLATED worlds), and side-panel scaffolding. Decision: do not migrate the code; rewrite from this spec when the extension is ready to land. The data model and message protocol will have evolved by then.

## 7. Tech stack (carried forward + new)

### Settled

- **Framework:** Next.js (static export), TypeScript, React
- **Storage:** IndexedDB via Dexie.js
- **Stats:** Pyodide/WASM (primary). The existing Lambda fallback is in tension with the privacy value prop — see §5.
- **UI:** **Tailwind CSS** (free / OSS tier). Design tokens from `design/Design System.html` go in `tailwind.config`. Component anatomy via `@apply` in `.css` files; avoid utility-class soup in JSX.
- **Charts:** Recharts
- **Testing:** Jest + SWC (`next/jest`), React Testing Library, `fake-indexeddb`
- **Power calc reference:** `statsmodels.stats.power.NormalIndPower`
- **CSV:** PapaParse, Web Worker for row-level aggregation (Welford's algorithm)
- **Deployment:** Cloudflare Pages + Workers
- **Content fingerprinting:** SHA-256 hash-chaining of committed fields

### New / changed from the predecessor

- **Bootstrap 5 + React Bootstrap → Tailwind.** Replaces the predecessor's UI stack entirely. Bootstrap-using components from `alphabeta-legacy` will need re-skinning when ported.
- **LLM integration:** Provider-adapter seam with capability negotiation at boot. Parse (light, local-viable) + Feedback (judgment, strong-model). `LLMProvider` interface surface TBD.
- **Lambda fallback under review:** Keep, but gate behind explicit consent — silent server-side compute violates the privacy positioning.

## 8. Personas

Three personas, in order of how much scaffolding they need:

1. **Newcomer** — learning experimentation; needs the most narration from the discipline layer.
2. **Practitioner** — knows the process; uses standard register; benefits from the commitment loop.
3. **Diagonal user** — the key persona. An *adjacent domain expert with a stats-shaped gap* (researchers, designers with deep domain expertise but limited statistical background). Not an experimentation veteran. Keeps authorship of the decision and leans on the method module for the math. This is the person the express lane, the Tweaks dial, and the scaffolded register are all calibrated around.

## 10. Evidentiary integrity context

A four-person core team has leadership's mandate to push back on undisciplined product management, including PMs actively misrepresenting study results. So integrity features are **product requirements, not nice-to-haves:**

- Timestamped pre-registration (before data exists)
- Immutable committed fields (edits create new versions, never overwrite)
- Content fingerprint proving what was committed
- Commitment-versus-outcome revisit structure
- Deviation logging in the user's words

This is the operational context that makes the lock, the fingerprint, and the deviation log non-negotiable rather than academic.

## 13. What NOT to do

- **Do not relitigate the core positioning.** "Can vs should," "discipline layer not stats tool," "bets not hypotheses" — these are settled.
- **Do not build compliance gates.** The discipline layer nudges and makes tradeoffs visible; it does not block workflows. Overrides are allowed but logged.
- **Do not add server dependencies to the static tier.** The pure-local, zero-server, no-accounts path must remain complete and functional. Backend features are additive, never required.
- **Do not silently ship data to a server.** Any server-side compute (Lambda, LLM API) requires explicit user awareness or consent.
- **Do not treat the fold-if as editable post-lock.** Edits create a new version. The lock is immutable.
- **Do not use `obsidian_patch_content` with heading targets.** It's unreliable. Use delete + recreate as the workaround for vault edits.
