Some bets rest on more than one prerequisite. Two prereqs (X and Y) both feed a downstream bet Z. The join type decides what "satisfied" means — and it isn't always obvious. Resolve the prereqs and watch how the join behaves; flip AND / OR to see the difference, and the case worth naming: a win that gets stranded.

Fan-in — when a bet needs more than one prerequisite

AND-join (default): all prereqs must win; any loss prunes Z. OR-join: either path is enough; Z prunes only if all lose. The join is a property of the edge, not a global rule — so it has to be set, not assumed.

The case to name: the orphaned win. Under AND, you can win X cleanly, but if Y loses, everything they jointly fed is pruned — and X's win is stranded. A real result, real effort, attached to a dead branch.

That's the argument for treating parallel prerequisites as a deliberate act, not a casual fan-out — the discipline layer makes the stranded win visible instead of letting it quietly absorb the cost. Under OR, a win is never orphaned; it satisfied the join on its own.
